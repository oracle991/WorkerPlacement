# PLAN.md — ワーカープレイスメント・プロトタイプ「辺境開拓団(仮)」

## コンセプト概要

プレイヤーは辺境の開拓団のリーダー。個性とステータスを持つワーカー(開拓民)を毎ラウンド、村周辺の「アクションスポット」に派遣する。スポットごとにイベントが発生し、**ワーカーのステータス + ランダム要素(ダイスロール)** で成否と結果の質が決まる。限られたラウンド内で村を発展させ、目標繁栄度に到達すればクリア。

ボードゲーム的な「置く→解決→補充」のループを、ソロプレイのデジタルゲームとして再構成する。

---

## 1. プレイヤーの目標

- **最終目標**: 制限ラウンド(標準 12 ラウンド = 4 季節 × 3 ラウンド)以内に、村の**繁栄度(Prosperity)を目標値(標準 30)まで上げる**。
- **中間目標**:
  - 食料・木材・鉱石・金貨の 4 資源をやりくりして村の施設を建設する(施設は繁栄度とパッシブ効果の源)。
  - ワーカーを増員・成長させ、行動の質と量を上げる。
  - ラウンドごとの食料消費(維持コスト)と冬季の環境悪化を乗り切る。
- プレイヤーが常に考えるのは「**誰を・どこに・どの順で置くか**」。ワーカーの得意分野とスポットの要求ステータスのマッチング、およびリスクの高いスポット(高リターン・低成功率)に誰を賭けるかの判断がゲームの核。

## 2. メインループ

1 ラウンドは以下の 4 フェーズで構成する。

```
[1] 準備フェーズ
    - 季節イベントの提示(例: 「豊作の兆し: 今ラウンド農場+1食料」)
    - 各スポットの今ラウンドのイベント内容とダイス目標値をプレビュー表示
[2] 配置フェーズ(プレイヤーの意思決定の中心)
    - 手持ちのワーカーを 1 人ずつスポットへドラッグ配置
    - スポットには定員あり(1〜2人)。配置済みスポットは選べない
    - 配置予定のワーカーとスポットの相性(成功率見込み %)を常時表示
    - 全員配置 or「このラウンドを終える」で確定
[3] 解決フェーズ(自動再生・演出)
    - スポット順に 1 箇所ずつイベント解決を演出付きで再生
    - 判定式: 2d6 + 対応ステータス + 個性補正 ≧ スポット難易度
      → 大成功 / 成功 / 失敗 / 大失敗 の 4 段階(達成差分で決定)
    - 結果に応じて資源獲得、ワーカーの経験値、負傷・疲労などを反映
[4] 維持フェーズ
    - ワーカー人数 × 1 の食料を消費(不足時は士気低下・離脱リスク)
    - 疲労回復、負傷者の回復判定
    - 繁栄度の更新と勝敗判定 → 次ラウンドへ
```

### ワーカーの設計

- **ステータス(1〜5)**: 筋力 / 器用 / 知恵 / 魅力 の 4 種。スポットごとに参照ステータスが決まっている(例: 森林伐採=筋力、市場=魅力)。
- **個性(Trait)**: 各ワーカーは 1〜2 個保持。判定や結果に条件付き補正を与える。
  - 例: 「怪力」= 筋力判定 +1 / 「臆病」= 難易度 9 以上のスポットで −1 / 「幸運」= 大失敗を 1 回だけ失敗に軽減 / 「働き者」= 疲労蓄積が半分 / 「社交的」= 同じスポットに相方がいると +1
- **状態**: 経験値(累積で成長しステータス+1)、疲労(連続稼働でペナルティ)、負傷(数ラウンド使用不可)。

### スポットの設計(初期 6 種 + 解禁 2 種)

| スポット | 参照ステータス | 主な報酬 | リスク |
|---|---|---|---|
| 農場 | 筋力 | 食料 | 低 |
| 森林 | 筋力 | 木材 | 低〜中(野獣遭遇) |
| 鉱山 | 筋力/器用 | 鉱石 | 中(落盤で負傷) |
| 市場 | 魅力 | 金貨・資源交換 | 低 |
| 工房 | 器用 | 施設建設(繁栄度) | — |
| 集会所 | 魅力 | 新ワーカー勧誘 | 中(外れワーカー) |
| 遺跡(第2季節で解禁) | 知恵 | 大量報酬・宝物 | 高(負傷・空振り) |
| 祠(第3季節で解禁) | 知恵 | 個性の付与・疲労全快 | 中 |

## 3. 入力と操作方法

- **対象デバイス**: PC ブラウザ(マウス操作)を第一。タッチは将来対応(操作体系はタッチ互換にしておく)。
- **基本操作**:
  - ワーカーのドラッグ&ドロップでスポットへ配置(クリック選択 → クリック配置の代替操作も用意)。
  - ワーカー / スポットのホバーで詳細ツールチップ(ステータス、個性、成功率見込み、報酬テーブル)。
  - 配置のやり直しは確定前なら自由(ドラッグで引き戻し)。
  - 画面下部の「ラウンド確定」ボタンで解決フェーズへ。解決演出はクリックでスキップ可。
- **キーボード**: 必須ではないが、Esc=選択解除、Space=演出スキップ、1〜9=ワーカー選択 を補助として実装。
- **UI 原則**: 判定に使う数値(2d6+X vs 難易度、成功率 %)は配置前に必ず見えるようにする。ランダム要素はあるが「知らされていない失敗」は作らない。

## 4. 勝利と失敗の状態

- **勝利**: 最終ラウンド終了時点で繁栄度 ≧ 目標値。達成ラウンドの早さ・残資源・ワーカー生存数でスコア/ランク(S/A/B/C)を算出。
- **失敗**:
  1. 最終ラウンド終了時に繁栄度が目標未満(標準の敗北)。
  2. ワーカーが 0 人になる(飢餓離脱・負傷全滅)。即時ゲームオーバー。
- **敗北しても学びがある設計**: リザルト画面で「どのラウンドで計画が崩れたか」を資源グラフで振り返れるようにする(プロトタイプでは簡易版でよい)。

## 5. 進行度・難易度

- **1 プレイ = 約 15〜20 分**(12 ラウンド)。ローグライト的な 1 セッション完結型。プロトタイプ段階ではメタ進行(周回強化)は持たない。
- **セッション内の難易度カーブ**:
  - 季節システム: 春→夏→秋→冬。冬は農場の産出半減・食料消費増で、事前の備蓄計画を強制する。
  - スポット解禁: 第 2・第 3 季節で高リスク高リターンのスポットが開き、選択肢とリスク判断が増える。
  - 繁栄度目標から逆算すると、序盤は経済基盤、中盤は施設建設、終盤は追い込みという自然な 3 幕構成になるよう数値を調整する。
- **難易度選択(プロトタイプ後半で)**: 目標繁栄度とイベントの過酷さを変えた Easy / Normal / Hard の 3 段階。
- **リプレイ性**: 初期ワーカー(3 人)のステータス・個性はシード付きランダム生成。イベントテーブルも抽選制。

## 6. ビジュアルの方向性

- **アートスタイル**: 2D 見下ろし気味の斜め俯瞰(ボードゲームの盤面をそのまま画面にした見た目)。温かみのあるファンタジー調ピクセルアート or フラットイラスト。プロトタイプでは**色分けした図形 + 絵文字/アイコンのプレースホルダで開始**し、後からアセット差し替え。
  - 無料アセット候補: Kenney (kenney.nl) の Board Game / Medieval パック、itch.io の CC0 タイルセット。
- **画面構成**:
  - 中央: 村マップ(スポットが配置されたボード)。
  - 下部: ワーカーの手札トレイ(カード風。顔・ステータス・個性を表示)。
  - 上部: 資源バー(食料/木材/鉱石/金貨/繁栄度)+ ラウンド/季節表示。
- **演出**: 解決フェーズのダイスロール演出(2 個のダイスが転がって止まる)を最重要演出とする。成功段階で色とエフェクトを変える(大成功=金、失敗=灰、大失敗=赤)。Phaser の Tween ベースで軽量に。
- **フィードバック**: 資源増減のフロートテキスト、ワーカーの表情アイコン変化(疲労・負傷)。

## 7. スタックとホスティングの前提条件

- **フレームワーク**: Vite + TypeScript + Phaser 3(確定)。
- **追加ライブラリ(最小限)**:
  - `seedrandom` または自前の mulberry32 実装 — シード付き乱数(リプレイ・デバッグ・テストの再現性に必須)。自前実装を推奨(依存ゼロ)。
  - `vitest` — ゲームロジックのユニットテスト。
  - 状態管理ライブラリは**入れない**。ゲームロジックは Phaser 非依存の純 TypeScript クラス群(`src/core/`)として書き、Phaser の Scene は描画と入力だけを担当する(テスト容易性とアセット差し替え耐性のため)。
- **データ駆動**: ワーカーの個性、スポット定義、イベントテーブル、季節修正は JSON/TS の定義ファイルに分離し、バランス調整をコード変更なしで行えるようにする。
- **ホスティング**: 完全静的サイト(サーバ・DB 不要)。`vite build` の成果物を GitHub Pages / Cloudflare Pages / Netlify いずれかに配置。セーブは当面 `localStorage`(中断再開のみ)。
- **対応環境**: デスクトップの Chrome / Edge / Firefox 最新版。解像度 1280×720 基準の固定アスペクト + Phaser の Scale.FIT。

## 8. マイルストーンの順序

| # | マイルストーン | 内容 | 完了条件 |
|---|---|---|---|
| M0 | プロジェクトセットアップ | Vite + TS + Phaser 3 雛形、vitest、ディレクトリ構成(core / scenes / data) | 空の盤面シーンがブラウザに表示される |
| M1 | コアロジック(描画なし) | ワーカー・スポット・判定式(2d6+ステータス+個性)・資源・ラウンド進行を純 TS で実装。シード付き乱数 | ユニットテストで 1 ゲームをヘッドレス完走できる |
| M2 | 盤面とプレースメント UI | プレースホルダ図形でボード表示、ドラッグ&ドロップ配置、ツールチップ、成功率表示 | マウス操作で 1 ラウンドの配置→確定ができる |
| M3 | 解決演出とフィードバック | ダイスロール演出、4 段階結果表示、資源フロートテキスト、演出スキップ | 配置→解決→維持の 1 ラウンドが通しで遊べる |
| M4 | ゲーム全体の完結 | 季節システム、スポット解禁、勝敗判定、タイトル/リザルト画面 | 12 ラウンドの 1 プレイが勝敗まで遊べる(垂直スライス完成) |
| M5 | コンテンツ拡充とバランス | 個性 10 種以上、イベントテーブル拡充、集会所での勧誘、負傷/疲労の調整 | 数回ランダムプレイして勝率がおよそ 40〜60% に収まる |
| M6 | 磨き込みとデプロイ | ビジュアルアセット差し替え、SE/BGM、難易度選択、localStorage セーブ、静的ホスティングへ公開 | 公開 URL で第三者がプレイできる |

**方針**: M1 でロジックを描画から完全に分離しておくことが最重要。M2 以降の UI 作業とバランス調整(データファイル編集)を並行して進められる。各マイルストーンは常に「動く状態」を保って積み上げる。
